# Android 自动化测试项目开发规则 v0.1

## 1. 核心原则

- 稳定性优先于功能数量
- 配置优先于硬编码
- 执行引擎优先于界面开发
- 可观测性是强制要求
- 所有动作必须可超时、可重试、可记录
- 默认按最小可行边界设计，避免过早复杂化

## 2. 文档主源与适用边界

- 首版范围、v1 必做/不做、默认运行交互和冻结技术决策以 [HIGH_LEVEL_DESIGN.md](../doc/HIGH_LEVEL_DESIGN.md) 为唯一主源
- DSL 字段语义、调度冲突处理、连续运行规则、运行记录结构和本地存储模型以 [DETAILED_DESIGN.md](../doc/DETAILED_DESIGN.md) 为唯一主源
- 本文档不重复定义产品范围和技术方案，只补充开发执行、测试、安全、数据落库和 Definition of Done 约束
- 本文档优先记录跨迭代稳定的工程与协作约束，避免重复承载某一版产品方案细节
- 若出现文档冲突：产品范围与技术方案以设计文档为准；研发流程、测试、日志、安全要求以本文件为准

## 3. 技术与架构规则

- 主语言使用 Kotlin
- 项目结构优先采用清晰分层，而不是按技术堆砌
- 任务能力通过声明式 DSL 扩展，不通过散落脚本扩展
- UI 层只负责配置、展示、触发和查看结果
- 执行引擎必须独立于 UI 层，可后台运行
- 高权限操作必须收敛到统一执行层
- 平台能力与外部系统能力必须收敛到统一适配层
- 可选能力必须通过统一能力接口暴露，并保持可插拔
- 调度器只负责触发，不负责具体动作执行
- 任务解析、任务执行、结果记录必须分离
- 实现验证范围必须与当前设计主源保持一致，不得自行扩大平台兼容范围

## 4. DSL 与执行规则

- 任务 DSL 结构、字段语义和执行默认值以 [DETAILED_DESIGN.md](../doc/DETAILED_DESIGN.md) 为唯一主源
- Parser、调度器和执行引擎的实现结果必须与设计文档中的字段定义、默认值和状态语义保持一致
- 坐标类动作只能作为兜底方案，不得替代已有可行的元素定位路径
- 所有执行实例都必须生成唯一运行编号
- 所有步骤都必须支持取消和超时
- 新增步骤类型时，必须同步补充参数校验、执行器、错误码和测试
- 新增或修改 DSL 语义前，必须先更新 [DETAILED_DESIGN.md](../doc/DETAILED_DESIGN.md)

## 5. 配置与数据规则

- 任务配置、凭据模型和运行记录结构以 [DETAILED_DESIGN.md](../doc/DETAILED_DESIGN.md) 为主源
- 配置必须可版本化，配置变更必须可追踪
- 任务配置需要有 schema 校验，不允许把任务核心配置散落到界面逻辑中
- 设备相关差异配置必须集中管理
- 敏感信息不得硬编码在任务配置、日志和诊断产物中
- Room Entity 不得直接暴露给 UI、调度器或执行引擎，必须经 Repository 转换后使用
- 涉及账号组顺序、连续运行游标和运行记录状态的多表写入必须通过事务完成

## 6. 日志与诊断规则

- 日志必须结构化
- 每次任务执行必须有完整链路记录
- 每个步骤都必须记录开始、结束、状态和关键参数摘要
- 失败时默认保留截图
- 感知类或识别类步骤需要保留中间诊断信息
- 错误码必须统一定义，禁止直接抛出无法分类的字符串错误
- 日志保留策略和日志级别必须可配置

## 7. 测试规则

- 核心模块新增代码覆盖率目标不低于 85%
- 任务解析器、调度器、执行引擎必须有单元测试
- 涉及平台能力、设备依赖能力或外部适配能力的模块必须有集成测试或设备测试
- 新增步骤类型必须覆盖成功、失败、超时三个场景
- 凭据读取和变量替换必须有单元测试
- 关键变更必须做真实设备验证
- 发布前必须至少完成一次 72 小时稳定性测试

## 8. 开发流程规则

- 所有开发工作必须关联需求文档和任务项
- 修改任务语义、调度语义、权限模型前必须先改文档
- 功能完成不等于工作完成，必须包含测试、日志和错误处理
- 不允许为了快速跑通而绕过统一执行与记录链路
- 不允许在页面代码中直接拼接任务流程逻辑
- 不允许将设备特定兼容逻辑散落在多个模块

## 9. 安全与权限规则

- 高权限能力只暴露给受控模块
- 任何高权限操作都必须有调用来源和执行记录
- 默认最小化权限申请
- 对外导出能力必须显式评审
- 运行环境检查失败时必须阻断执行并给出明确原因

## 10. Definition of Done

- 需求已明确并更新文档
- 设计边界已明确
- 代码实现完成
- 单元测试通过
- 必要的设备测试通过
- 错误码和日志完整
- 截图和诊断链路完整
- 代码评审通过
- 未引入已知稳定性回退

## 11. 首批必须冻结的约束

- 任务采用声明式 DSL，不用录制回放做核心方案
- 执行引擎统一管理步骤执行生命周期
- 调度器与执行器解耦
- 统一错误码与统一执行结果结构
- 先验证核心真实场景稳定运行，再扩展高级能力
